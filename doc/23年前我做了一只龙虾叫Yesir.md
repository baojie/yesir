# 23年前我做了一只龙虾叫Yesir

2003年12月，我在Iowa State University读博，写了一个Java程序叫**Yesir! Information Agent**。那年我给它的副标题是"语义网关"（Semantic Gateway）。包名叫`com.semanticwww`。

2026年1月，一只叫OpenClaw的龙虾登上了GitHub星标榜首，24.8万星，超过了Linux。

我翻出23年前的代码，骨架竟然长得差不多。但一个活了，一个没活。这篇文章想搞清楚的不是"我多有远见"，而是：**同样的直觉，为什么在2003年长不出来，在2026年却长疯了？**

## 一、两只龙虾的共同蓝图

Yesir的核心架构只有三层：

```
数据源 (Importer) → 统一信息单元 (Meme) → 数据目标 (Exporter)
```

翻译成今天OpenClaw的话术：

```
外部服务接入 → Agent上下文 → 外部服务输出
```

两者解决的是同一个问题：**人的信息散落在无数个互不相通的平台上，需要一个代理把它们连起来。**

2003年，这些平台是RSS订阅、POP3邮箱、NNTP新闻组、Google搜索、BBS论坛。2026年，它们变成了WhatsApp、Telegram、Discord、Gmail、日历、文件系统。名字换了，问题没变。

Yesir的`Gateway`类组合一个Importer和一个Exporter，支持定时自动执行、手动触发、循环调度。OpenClaw的bot同样在本地持续运行，轮询各个服务，自主决定何时行动。Yesir有`TaskEngine`做线程池和优先级队列调度；OpenClaw有事件循环和异步任务链。底层机制不同，但调度模型的意图完全一致。

那时候我就管这个系统叫**agent**。`About`对话框里写的是"Yesir! information agent"。这不是后来的追溯命名——2003年的代码里就是这么叫的。Agent这个词在AI领域的历史远比今天的热潮古老得多。

## 二、为什么最核心的数据单元叫Meme

Yesir里流转的所有信息都被统一为一个类：`Meme`。

这个命名不是随意的。代码注释里写得很清楚：

> *meme - An element of culture that may be considered to be passed on by non-genetic means, esp. imitation.*
>
> *meme is spreading on the internet and the brains of us people. Hopefully, software can speed up its evolution speed, just like birds help in spreading seeds.*

Meme这个概念来自Richard Dawkins 1976年的《自私的基因》。他造了这个词来类比基因——基因是生物信息的复制单元，meme是文化信息的复制单元。一段旋律、一个口号、一种时尚、一条新闻，都是meme。它们在人脑之间传播、变异、竞争、进化，和基因在身体之间的传播遵循相似的动力学。

这恰好精确描述了Yesir要处理的东西：一封邮件、一条RSS条目、一篇新闻帖子、一个搜索结果——它们都是在互联网上流动的文化信息单元。它们从一个平台被"导入"，经过转换，再"导出"到另一个平台。这个过程本身就是meme的传播和变异。

所以我没有把这个类叫`Message`或者`Item`或者`Record`。**叫Meme，是因为这个系统本质上是一台meme传播加速器。** 它不只是搬运数据，它是在帮助信息在不同的生态位之间扩散和适应。

在1.0.0版本里这个类还叫`Mail`——那时候只能处理邮件。当我把它泛化为能处理RSS、新闻、搜索结果等所有信息类型时，需要一个更抽象的名字。Meme完美地表达了这个抽象：**不管载体是什么，它承载的都是一段在传播中的文化信息。**

具体实现上，Meme基于Dublin Core元数据标准和RSS 1.0规范，包含title、creator、date、content、category、source等15个字段。这套字段从邮件到新闻到搜索结果都能映射上去——这正是Dublin Core作为"元数据的元数据"被设计的初衷。

## 三、从语义网到大语言模型：建模方法变了，问题没变

2003年我用的武器是**语义网（Semantic Web）**的那套东西。

那个年代Tim Berners-Lee的愿景正热：给互联网上的每一份信息加上机器可读的语义标注，让软件能"理解"数据的含义。Dublin Core定义了15个核心元数据元素（标题、创建者、主题、描述……），RSS用RDF（资源描述框架）来结构化新闻订阅，OWL本体语言试图形式化地描述世界知识。

Yesir的做法是：用Dublin Core + RSS作为统一的语义框架，把不同来源的数据对齐到同一套元数据模型上。一封POP3邮件的`From`字段映射为Meme的`author`，一条RSS订阅的`<dc:creator>`也映射为`author`。通过这种显式的语义对齐，不同格式的数据就能在同一个管道里流转。

这种方法的核心假设是：**如果我们能把知识的结构显式地编码出来，机器就能理解和处理它。**

23年后，OpenClaw用的武器换成了**大语言模型（LLM）**。

它不需要Dublin Core，不需要RDF，不需要本体。它把一封邮件的原始文本丢给Claude或GPT，模型直接"读懂"了这是谁发的、关于什么主题、需要做什么回应。语义对齐不再由程序员手工编写映射规则，而是由模型在海量文本训练中隐式习得。

| 维度 | 语义网方法 (2003) | LLM方法 (2026) |
|------|-------------------|-----------------|
| 知识表示 | 显式本体、元数据标准 | 神经网络参数中的隐式知识 |
| 语义对齐 | 手工编写字段映射 | 模型自动理解 |
| 扩展新数据源 | 写新的Importer类+映射代码 | 写一段prompt描述 |
| 处理非结构化数据 | 几乎无能为力 | 天然擅长 |
| 可解释性 | 完全透明，每步可查 | 黑箱，难以审计 |
| 容错性 | 格式不对就崩溃 | 格式乱七八糟也能凑合 |

但如果你退后一步看全景，这两种方法试图解决的核心任务惊人地一致：

1. **感知**：从异构数据源获取信息
2. **理解**：把不同格式的信息转化为统一表示
3. **行动**：把处理后的信息投递到目标平台

感知-理解-行动。这就是agent的经典循环。2003年如此，2026年如此。

## 四、那些没变的东西

翻看23年前的代码，有些设计决策在今天看来依然完全成立：

**插件架构。** Yesir用Java的ClassLoader动态加载Importer和Exporter类，用户可以在运行时注册新的插件。OpenClaw用的是模块化的服务集成。本质都是同一个工程直觉：数据源会不断增加，系统必须是开放的。

**本地运行。** Yesir是一个本地Swing应用，所有数据在用户机器上处理。OpenClaw同样强调本地部署，配置和交互历史存储在本地。这不是巧合——当一个agent要接触你的邮箱、日历和聊天记录时，把数据留在本地是最起码的信任边界。23年前我这么想，今天OpenClaw的安全争议也恰恰围绕这一点。

**定时调度。** Yesir的Gateway用javax.swing.Timer做周期执行，支持"执行N次后停止"和"永远循环"两种模式。OpenClaw的bot同样持续运行、周期轮询。人类的信息流是不间断的，agent也必须是不间断的。

**统一数据模型。** Meme就是Yesir版本的"Agent上下文"。不管信息从哪里来、到哪里去，中间必须有一个规范化的表示层。没有这一层，每加一对源和目标就要写一套转换代码，复杂度会是O(n²)。有了统一模型，复杂度降为O(n)。这是最基础的软件工程，但23年来没有过时。

## 五、那些变了的东西

当然，变化是巨大的。

最根本的变化是：**2003年的agent是刚性的，2026年的agent是柔性的。**

Yesir的每一条数据通路都是程序员预先定义的。RSS进来的数据一定映射到这些字段，邮件进来的数据一定映射到那些字段。如果有一封邮件的格式稍微非标，解析就会失败。系统的能力上限等于程序员预见到的所有情况。

OpenClaw背后的LLM让这个等式变了。用户用自然语言说"帮我把邮件里的会议邀请提取出来加到日历上"，系统不需要预先定义"会议邀请"的格式规范，模型能从邮件文本中直接识别出时间、地点、参与者。**从规则驱动变成了理解驱动。**

这就是为什么语义网的宏大愿景最终没有实现——不是方向错了，是方法太硬了。要求全世界的信息发布者都遵循统一的元数据标准，这在工程上是不现实的。人类产生信息的方式是混乱的、非结构化的、不断变化的。LLM绕过了这个困境：它不要求数据是结构化的，它自己来做结构化。

但这里面有一个讽刺：**LLM做的事情，恰恰就是语义网想做的事情——只不过用了完全不同的路径。** 语义网说"让我们给数据加上语义标签"，LLM说"让我直接从数据中读出语义"。目的地相同，一个修路，一个飞过去了。

## 六、代码里的Agent设计直觉——23年后依然有效

重新翻读2003年的代码，让我惊讶的不是哪些设计过时了，而是有多少工程直觉在23年后的agent系统里依然是标配。

**反饥饿调度。** `TaskEngine`里有一个手写的三级优先级队列（HIGH/MEDIUM/LOW），但它不是简单地永远先执行高优先级任务。每次取出一个任务时，都会把一个低优先级任务提升到中优先级，一个中优先级任务提升到高优先级。这防止了某些Gateway永远得不到执行。今天的任务调度框架里这叫"aging"策略，是每个agent编排系统都要解决的公平性问题——当你的agent同时连接了邮箱、日历和聊天，你不能让日历同步永远排在邮件之后。

**弹性资源池。** `TaskEngine`的工作线程从5个起步，积压时每2秒扩一个（最多30），空闲5秒缩一个（最少3个）。今天Kubernetes的HPA、AWS Lambda的并发伸缩，用的是同一种直觉。Agent同时处理多少任务是动态变化的，资源分配必须跟着变。

**零侵入的插件持久化。** `BeanUtils`用Java反射自动遍历Importer/Exporter的所有JavaBean属性，把它们序列化成XML。插件开发者不需要写任何序列化代码——只要遵循JavaBean规范，配置的保存和恢复就自动生效。今天你给Claude Code写一个MCP server，只要遵循Tool定义规范，接入也是自动的。**"约定优于配置"不是Rails发明的，它是所有好的插件系统的共识。**

**原子写入。** 配置保存用临时文件+改名，崩溃后自动恢复。Agent是长期运行的服务，任何时刻都可能被中断。持久化必须是原子的，否则一次意外重启就可能丢失所有Gateway配置。今天每一个agent框架都要解决这个问题：如何保证agent的状态在崩溃后可恢复。

**协议兼容的韧性。** RSS标准有四个互不兼容的版本（0.91、0.92、1.0/RDF、2.0），`RSSXMLParser`全部支持，通过检测命名空间自动判断版本。真实世界的数据源不会按标准来——这是每一个agent开发者都会学到的教训。今天的agent对接各种API时面对的版本碎片化问题，和2003年的RSS乱局本质上没有区别。

**编码地狱。** `ChineseUtils`处理GBK、BASE64中文、ISO-8859-1转码。连javadoc生成的HTML都是GBK编码的——`<META charset="GBK">`，不是UTF-8。2003年的互联网没有统一编码，每个数据源的字符集都可能不同。这是语义网理想化假设在现实面前碰壁的缩影，也预示了今天agent处理多语言、多格式数据时面临的挑战——LLM帮你绕过了编码问题，但多模态数据的格式差异依然需要工程层面的适配。

**安全边界。** `RSSXMLFactory`支持HTTPS和HTTP Basic认证，拒绝加载外部DTD（防XXE攻击）；`Validator`拒绝`postmaster@`和`root@`等高危地址。Agent天然要接触外部数据源，安全顾虑从第一天就存在。OpenClaw因为权限过大引发的安全争议，不过是同一个问题在23年后的放大版。**任何能读你邮件的agent，都必须回答"凭什么信任你"这个问题。**

**Google SOAP API。** `GoogleImporter`调用Google的SOAP搜索API——这个API在2006年就关闭了。代码里硬编码了一个API key。这大概是最有年代感的一段代码：那时候Google还开放搜索接口让你免费调用。但它揭示的问题今天更尖锐：agent依赖的外部API随时可能关闭、限流或改版。**插件架构的真正价值，就是让agent在某个API消失时还能活下去。**

## 七、为什么Yesir没活下来

说实话，Yesir没做下去不是因为"生不逢时"这么浪漫的原因。是好几个很具体的问题叠在一起，让它注定长不大。

**第一，没有用户场景兜底。** 2003年的信息焦虑还没到今天这个程度。普通人的数字生活就是一个邮箱、几个网页收藏夹、偶尔看看BBS。RSS订阅是极客的玩具，NNTP新闻组更是小众中的小众。Yesir能连接的那些数据源，大部分人根本不用。**一个整合工具的价值上限，取决于用户有多少东西值得整合。** 2003年答案是：不多。

**第二，每加一个数据源都是苦力活。** 写一个新的Importer意味着：研究协议文档、处理认证、解析格式、做字段映射、处理编码、写GUI配置面板。javadoc的类继承树忠实地记录了这个工作量：5个Importer、5个Exporter，每一个都配着自己的SetupDlg配置对话框——总共10个手写的Swing面板。一个Pop3Importer从头到尾大概2000行代码。邮件发送那个Exporter甚至类名都拼错了：`SmtpEporter`，少了个x，到今天都没改过来。2003年12月13日，周六上午，javadoc生成的时间戳。一个博士生课余时间能写几个Importer？我写了5个就到极限了。没有社区，没有生态，全靠一个人肝。

**第三，语义网的"先结构化再处理"路线走不通。** 这不是事后诸葛亮——当时就已经感觉到了。每对接一个新数据源，光是把字段对齐到Dublin Core就要花大量时间，而且总有些信息塞不进那15个字段里。一封邮件的附件怎么映射？一条RSS里嵌的播客音频怎么表示？标准覆盖不了的地方，系统就是个瞎子。

**第四，平台API极不稳定。** Google SOAP API 2006年就关了，很多RSS源朝生暮死。Agent依赖的数据源是别人的地盘，别人随时可以拆桥。这个问题今天依然存在，但2003年更严重——那时候没有OAuth标准，没有API版本管理的行业共识，接口说变就变。

**第五，也是最根本的——读博要毕业。** Yesir是一个side project，不是我的研究方向。学位论文要的是本体映射的理论工作，不是一个Java桌面应用。当学术压力上来的时候，Yesir就被搁下了。这大概是大多数博士生side project的宿命。

回头看，Yesir的问题不在于设计错了，而在于**它试图用手工劳动去填一个需要生态系统才能填的坑**。一个人写5个Importer是极限，但用户需要500个。没有社区贡献、没有插件市场、没有网络效应，一只孤独的龙虾翻不起浪。

## 八、为什么今天这只龙虾能火

OpenClaw在2026年1月爆发不是偶然。它踩中了好几个同时成熟的条件。

**LLM把Importer的开发成本打到接近零。** 这是最关键的变化。2003年写一个Pop3Importer要2000行Java代码，2026年对接一个新服务可能只需要几十行代码加一段prompt。LLM承担了"理解数据格式"这个最脏最累的活。这意味着一个小团队就能快速覆盖几十个服务，不用像Yesir那样一个个手写解析器。**工程量级的变化带来了产品可能性的变化。**

**数字生活的复杂度到了临界点。** 今天一个普通知识工作者同时使用的通讯和协作工具超过10个——Slack、Teams、Discord、WhatsApp、Telegram、微信、Gmail、日历、Notion、飞书……信息碎片化的痛感是2003年的十倍不止。需求终于追上了解决方案。

**本地优先的信任模型恰好契合隐私焦虑。** 2003年没人在意数据隐私，大家巴不得把所有东西都放网上。2026年人们经历了无数次数据泄露，GDPR也教育了市场。OpenClaw强调本地运行、数据不上传，正好打中了这个时代情绪。

**开源社区的成熟度完全不同了。** 2003年开源还是理想主义者的事业。2026年GitHub有1亿+开发者，npm/pip/cargo生态意味着你可以站在无数人的肩膀上。OpenClaw能火，一半功劳是它自己的设计，一半功劳是整个开源基础设施。Yesir那个年代，你想找一个好用的RSS解析库都费劲。

**API经济的成熟。** OAuth 2.0、REST API设计共识、GraphQL、Webhook——这些基础设施在2003年要么不存在，要么处于萌芽期。今天对接一个SaaS服务有标准化的路径，而不是像2003年那样面对每个服务都要从零开始摸索协议。

用一句话总结：**2003年的问题是真问题，但土壤不够肥。** LLM提供了理解能力，开源生态提供了工程杠杆，数字生活的复杂度提供了用户需求，隐私焦虑提供了产品叙事。这四样东西在2026年同时就位，龙虾就活了。

不是谁的设计更聪明——是季节到了。

## 九、如果在2003年有LLM

有时候我会想，如果2003年我就能用上今天的语言模型，Yesir会是什么样子。

答案可能是：Meme类不需要15个显式字段了，取而代之的是一段原始文本加上模型按需提取的结构化信息。Importer不需要为每种格式写解析器了，把原始数据丢给模型就行。Exporter也不需要手工拼装输出格式了，告诉模型"把这个信息写成一封邮件"就够了。

想想看具体会省掉什么。`MailParser`里有500多行代码在处理MIME多部分消息的递归解析、附件提取、邮件标志位（ANSWERED、DELETED、DRAFT、FLAGGED……）的逐个判断。`RSSXMLParser`里有400多行代码在对付四个版本的RSS格式差异和命名空间兼容。`ChineseUtils`里在和GBK、BASE64、ISO-8859-1编码搏斗。这些代码每一行都是在手工告诉机器"这个字段的意思是什么"——而LLM天生就知道。

更深一层：Yesir的五个Example工作流都是硬编码的。"RSS → Blank"、"Google → RSS"、"POP3 → RSS"——每一条数据通路都是程序员预先定义的组合。如果有LLM，用户可以直接说"帮我把Iowa天气预报的RSS订阅内容每天早上发到我邮箱"，agent自己就能组装出这条通路。**工作流从预定义变成了按需生成。** 这正是今天OpenClaw最核心的能力差异。

甚至`GatewayCreateDlg`那个让用户从下拉框选Importer和Exporter的界面也不需要了。用户不需要知道什么是Importer、什么是Exporter、什么是Gateway——这些都是程序员的抽象。用户只需要说出他想要什么，agent自己知道该调用哪个组件。**LLM消灭了程序员抽象和用户意图之间的翻译层。**

但Gateway的架构不会变。TaskEngine的调度逻辑不会变。插件注册机制不会变。本地运行的原则不会变。

**因为这些东西不是关于"如何理解信息"的——它们是关于"如何搬运信息"的。** 不管理解信息的方式从语义标注变成了神经网络，信息本身还是需要从A点到达B点，还是需要一个代理来执行这个过程，还是需要调度、持久化、容错这些工程基础设施。理解层可以被LLM取代，工程层不会。

反过来想也成立：如果今天的OpenClaw穿越回2003年，没有LLM可用，它必须退化成什么样子？大概就是Yesir的样子——手写解析器、显式字段映射、预定义工作流、下拉框选择插件。**Yesir就是没有LLM的OpenClaw，OpenClaw就是有了LLM的Yesir。** 两只龙虾之间隔的不是设计思想的鸿沟，而是一个语言模型的距离。

## 十、知识表示：agent智能性的真正来源

有一个更深层的线索值得追溯：**是知识表示赋予了agent智能性，而不是反过来。**

Yesir的Meme类基于Dublin Core元数据规范——它定义了title、creator、date、subject等描述信息资源的标准字段。Dublin Core本身只是一套元数据约定，但它背后的思路是本体论的：**用一套共享的概念框架来统一描述不同来源的信息。**

真正赋予Yesir智能性的，是这套概念框架所实现的**本体映射（ontology mapping）**。当系统知道一封邮件的`From`字段和一条RSS的`dc:creator`元素在语义上等价时，它才能把两者放进同一个管道处理。`Pop3Importer`把邮件头解析为Meme的`author`，`RSSImporter`把`<dc:creator>`也映射为`author`，`NewsImporter`把新闻组消息的发件人同样映射为`author`——三种完全不同的数据格式，通过本体映射汇聚到同一个语义槽位。

这种映射不是简单的字段改名。邮件的`From`可能包含"Jie Bao <baojie@iastate.edu>"这样的格式，RSS的`dc:creator`可能只是纯文本"Jie Bao"，新闻组的发件人可能是"baojie@iastate.edu (Jie Bao)"。每个Importer在做映射时，都在执行一次微型的语义对齐：从不同的语法表示中提取出相同的语义含义。没有这层对齐，Importer吐出来的就是一堆无法互操作的字节流，Gateway只是一个哑巴管道。**本体映射把数据变成了知识，知识让agent有了判断力。**

但23年间发生了一件吊诡的事：**知识在演进过程中一度被丢掉了。**

从语义网到知识图谱的过渡中，"知识"这个词被保留了，实质却被掏空了。知识图谱（Knowledge Graph）听起来很美，但大多数工业界的知识图谱其实只有"图谱"没有"知识"——它们本质上是结构化的数据库，存的是实体和关系的三元组，而不是关于世界的可推理的语义约定。Google的Knowledge Graph能告诉你"北京是中国的首都"，但它不知道"首都"意味着什么——不知道首都通常是政治中心、不知道迁都意味着什么、不知道"临时首都"和"正式首都"的区别。图谱记录了事实，但丢失了理解事实所需要的概念框架。

语义网时代的OWL本体试图用形式逻辑来编码这些概念框架——类的层次、属性的约束、推理的规则。这是真正的"知识"，但它太重了，写起来太难了，维护起来更难。于是工业界做了一个务实的选择：放弃本体，只留三元组。知识图谱因此得以大规模部署，但代价是智能性的退化。

**现在，skill把知识加了回来。**

今天的AI agent用**skill**来表达知识——但这次知识的形式既不是形式逻辑（太重），也不是纯三元组（太薄），而是一种**结构化自然语言（Structured Natural Language, SNL）**。

看一个Claude Code的skill定义：它用自然语言描述了"什么时候触发"、"怎么执行"、"输出什么格式"。看OpenClaw的一个服务集成：它用自然语言+JSON Schema描述了"这个API接受什么参数"、"返回值代表什么含义"、"在什么场景下应该调用"。这些都不是形式逻辑，但它们携带了丰富的语义信息——因为LLM能"读懂"自然语言中的语义，形式逻辑不再是传递知识的唯一通道。

这是知识表示历史上的一个重要转折：

| 时代 | 知识表示载体 | 形式 | 知识的厚度 |
|------|-------------|------|-----------|
| 1990s | Cyc知识库 | 形式逻辑断言 | 极厚：概念、规则、常识推理 |
| 2003 | Dublin Core / RDF / OWL | 元数据标准+本体 | 中等：语义约束+推理规则 |
| 2012 | Knowledge Graph | 三元组 (SPO) | 极薄：只有事实，没有理解 |
| 2026 | Skill / Tool / Prompt | 结构化自然语言 (SNL) | 回归丰富：语义由LLM补全 |

**从Cyc到OWL，知识从"厚"走向"薄"，因为形式化越重，维护成本越高。从知识图谱到Skill，知识又从"薄"走回"厚"，因为LLM让自然语言也能承载精确的语义。** 这是一个螺旋式的回归：知识的丰富度回来了，但载体从逻辑公式变成了自然语言。

Skill就是新时代的本体。它告诉agent这个世界有哪些概念、这些概念之间是什么关系、对这些概念可以执行哪些操作——和OWL做的是同一件事，只不过用的是LLM能理解的结构化自然语言，而不是人类都难以阅读的描述逻辑。

这也是为什么"纯LLM"并不足以构成一个真正有用的agent。LLM有理解力，但它不知道你的Gmail密码，不知道你公司的日历API端点，不知道你的Slack workspace ID。这些具体的、情境化的知识必须通过skill注入给agent。LLM提供了理解能力，skill提供了领域知识，两者缺一不可。

Yesir的Meme类是我在2003年对这个问题给出的一个朴素答案：用Dublin Core本体来规范化信息的语义结构。中间经过了知识图谱的"去知识化"弯路，今天的skill又把知识带了回来——只不过换了一种LLM能消化的形式。那个根本问题——**如何表示知识，以使agent能够智能地行动**——一天都没有改变过。

从Cyc到Dublin Core，到知识图谱，再到Skill，知识表示走了一个完整的螺旋。知识先是被形式化，然后被简化到只剩骨架，现在又以自然语言的形式复活。知识表示一直是agent的灵魂。没有它，agent只是一堆管道工具；有了它，agent才有了理解和决策的能力。23年前如此，今天依然如此。

## 十一、下一只龙虾

翻完旧代码，想清楚了两只龙虾的异同之后，真正有意思的问题是：**下一只长什么样？**

OpenClaw解决了"连接"的问题——它能高效地把各种服务串起来。但它本质上还是一只反应式的龙虾：用户说一句，它做一步。下一代agent要解决的是**主动性**的问题：不是用户告诉agent该做什么，而是agent观察到模式后自己提出建议。

Yesir的`TaskEngine`有一个朴素的调度循环：定时拉取、定时推送。OpenClaw加上了LLM的理解力。但两者都缺少一个东西：**记忆和学习**。agent用了三个月，它应该比第一天更了解你——知道你周一早上先看邮件再看日历，知道老板发的消息你总是秒回但newsletter你攒着周末看，知道你每次收到会议邀请都会先查日程冲突。这些行为模式今天的agent都不会去学。

从架构上看，Yesir的三层模型需要加第四层：

```text
数据源 → 统一信息单元 → 数据目标
                ↕
           用户模型 (记忆/偏好/模式)
```

这不是一个全新的想法——推荐系统早就在做用户建模了。但agent场景下的用户模型和推荐系统不同：它不是在优化点击率，而是在学习**你的工作方式**，然后把自己变成你工作方式的延伸。

另一个方向是**agent之间的协作**。Yesir是一只独立的龙虾，OpenClaw也是。但现实场景里，你可能需要一只龙虾管邮件，另一只管代码，第三只管日程，它们之间需要互通信息、协调行动。MCP（Model Context Protocol）正在往这个方向走，但还处于非常早期的阶段。agent的互操作性问题，和2003年数据源的互操作性问题，又是同一个问题的不同投影。

还有一件事值得做：**把skill的创建门槛再降一个量级。** 今天写一个MCP server或者一个Claude Code skill，还是需要程序员。如果普通用户能用自然语言描述"我希望agent在收到客户邮件时自动查CRM里的客户信息，然后帮我起草回复"，系统自动把它编译成一个skill——那agent的能力扩展就真正民主化了。从Yesir的手写Importer，到OpenClaw的代码化集成，到未来的自然语言skill定义，这是同一条路线的延伸。

23年前的那只龙虾和今天的这只，长得不一样，但它们在同一片海里觅食。下一只龙虾的骨架大概还是一样的——感知、理解、行动——但它会长出记忆、长出主动性、长出和其他龙虾协作的能力。
